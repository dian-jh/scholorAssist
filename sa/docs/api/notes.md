# 笔记/标注模块 API 文档

## 📖 概述

笔记模块主要用于 **PDF 阅读器的高亮与批注**。 与传统“记事本”不同，本模块的笔记是依附于文档特定位置（坐标）的。前端通过记录用户划线的坐标 (`coord`) 和选中的原文 (`referenceText`)，配合用户的评论 (`content`) 形成一条完整的标注记录。

## 🔗 基础信息

- **基础 URL：** `/api/notes`
- **服务端口：** `10170` (参考 `application.yaml`)
- **认证方式：** Header 中需携带 Token
- **内容类型：** `application/json`

## 📋 接口列表

### 1. 创建笔记/标注

**接口名称：** 创建笔记

**接口地址：** `/api/notes`

**请求方式：** `POST`

#### ⚙️ 功能说明

当用户在 PDF 阅读器中选中一段文字并点击“高亮”或“写笔记”时调用。

#### 📥 请求参数 (Body)

对应后端类：`NoteCreateRequest`

```
{
  "documentId": "1865230658941235200",
  "content": "这段话是Transformer的核心思想",
  "referenceText": "Attention is all you need...",
  "coord": "{\"page\": 1, \"rect\": [100, 200, 300, 400]}",
  "color": "#FFCC00",
  "tags": ["重点", "AI原理"]
}
```

| 参数名            | 类型   | 必填 | 说明     | 设计思考 (Why)                                               |
| ----------------- | ------ | ---- | -------- | ------------------------------------------------------------ |
| **documentId**    | string | ✅ 是 | 文献 ID  | 关联具体文档。                                               |
| **content**       | string | ❌ 否 | 笔记内容 | 用户输入的评论。如果只是单纯高亮不写字，此字段可为空。       |
| **referenceText** | string | ❌ 否 | 引用原文 | **关键字段**。用于后续 RAG 检索或 AI 问答时的上下文引用。    |
| **coord**         | string | ✅ 是 | 坐标信息 | **关键字段**。通常存储 JSON 字符串。前端 pdf.js 需要根据此信息在正确的位置重绘高亮框。建议包含页码信息。 |
| **color**         | string | ❌ 否 | 高亮颜色 | 例如 `#FFFF00`。支持用户自定义高亮颜色。                     |
| **tags**          | array  | ❌ 否 | 标签列表 | 支持多标签管理。                                             |

#### 📤 响应参数

返回创建成功的笔记 ID。

```
{
  "code": 200,
  "message": "操作成功",
  "data": "186999..."
}
```

### 2. 查询笔记列表 (搜索)

**接口名称：** 检索笔记

**接口地址：** `/api/notes/list`

**请求方式：** `GET`

#### ⚙️ 功能说明

用于在阅读页右侧边栏加载当前文档的所有笔记，或者在“我的笔记”页面搜索笔记。 **注意：** 为了支持复杂查询参数，此处使用了 POST 请求。

#### 📥 请求参数 (Body)

对应后端类：`NoteQueryRequest`

```
{
  "documentId": "1865230658941235200",
  "keyword": "Attention",
  "tags": ["重点"]
}
```

| 参数名         | 类型   | 必填 | 说明                                                   |
| -------------- | ------ | ---- | ------------------------------------------------------ |
| **documentId** | string | ❌ 否 | 如果传了，查该文档下的笔记；不传则查该用户的所有笔记。 |
| **keyword**    | string | ❌ 否 | 模糊搜索 `content` 或 `referenceText`。                |
| **tags**       | array  | ❌ 否 | 筛选包含指定标签的笔记。                               |
| **page**       | int    | ❌ 否 | 页码 (默认1)                                           |
| **pageSize**   | int    | ❌ 否 | 页大小 (默认10)                                        |

#### 📤 响应参数

对应后端类：`NoteResponse`

```
{
  "code": 200,
  "data": {
    "total": 5,
    "records": [
      {
        "id": "186999...",
        "documentId": "1865230658941235200",
        "content": "这段话是Transformer的核心思想",
        "referenceText": "Attention is all you need...",
        "coord": "{\"page\": 1, \"rect\": [...] }",
        "color": "#FFCC00",
        "tags": ["重点"],
        "createTime": "2025-12-20 10:00:00"
      }
    ]
  }
}
```

### 3. 修改笔记

**接口名称：** 更新笔记信息

**接口地址：** `/api/notes`

**请求方式：** `PUT`

#### ⚙️ 功能说明

用户修改笔记内容、颜色或标签。**注意：通常不允许修改 `documentId` 或 `coord` (坐标)，因为那意味着是另一条笔记了。**

#### 📥 请求参数 (Body)

对应后端类：`NoteUpdateRequest`

```
{
  "id": "186999...",
  "content": "修改后的想法...",
  "color": "#00FF00",
  "tags": ["复习"]
}
```

| 参数名      | 类型   | 必填 | 说明                    |
| ----------- | ------ | ---- | ----------------------- |
| **id**      | string | ✅ 是 | 笔记 ID                 |
| **content** | string | ❌ 否 | 新的内容                |
| **color**   | string | ❌ 否 | 新的颜色                |
| **tags**    | array  | ❌ 否 | 新的标签列表 (全量覆盖) |

#### 📤 响应参数

```
{
  "code": 200,
  "message": "操作成功",
  "data": true
}
```

### 4. 删除笔记

**接口名称：** 删除笔记

**接口地址：** `/api/notes/{id}`

**请求方式：** `DELETE`

#### ⚙️ 功能说明

删除一条笔记/高亮。

#### 📥 请求参数 (Path)

| 参数名 | 类型   | 必填 | 说明    |
| ------ | ------ | ---- | ------- |
| **id** | string | ✅ 是 | 笔记 ID |

#### 📤 响应参数

```
{
  "code": 200,
  "message": "操作成功",
  "data": true
}
```

## 💡 开发者备忘 (Student Notes)

### 1. 为什么没有 Title (标题) 字段？

在学术文献阅读场景中，笔记往往是针对“某句话”的随想（Annotation），而不是一篇独立的文章。因此，`referenceText`（引用的原文）其实充当了“标题”的作用，帮助用户快速回忆起笔记的上下文。

### 2. coord 字段怎么存？

后端定义为 `String` 类型，但建议前端存入 **JSON 字符串**。 推荐结构（适配 pdf.js）：

```
{
  "pageIndex": 0,  // 页码，注意大部分插件从0开始，还是从1开始，需前后端统一
  "rects": [       // 高亮区域通常可能跨行，所以是矩形数组
    { "x": 100, "y": 200, "w": 50, "h": 10 },
    { "x": 50,  "y": 210, "w": 30, "h": 10 }
  ]
}
```

后端只负责存取，不负责解析该字段的业务含义。

### 3. ID 精度问题

再次强调，后端使用 `SnowflakeIdGenerator` 生成的是 19 位 `Long`。前端 JS 的 `Number` 类型最大只支持 16 位精度。 **务必确保后端在序列化 JSON 时将 ID 转为 String**（检查 `Result` 或 `NoteResponse` 中的 `@JsonSerialize` 配置），否则前端拿到的 ID 尾数会变成 0，导致删除/修改失败。
